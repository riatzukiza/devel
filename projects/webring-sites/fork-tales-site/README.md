# fork-tales-site

Fork Tales is now split cleanly:

- the **frontend stays static** and keeps the haunted retro shell in `src/`
- the **backend is a real FastAPI app** in `fork_tales_api/`
- the **site snapshot** is still generated by `build_site.py`
- the **runtime deploy** is containerized through `compose.yaml`
- the **Promethean deploy flow** lives in `scripts/` and GitHub Actions

## Architecture

### Frontend

Unchanged on purpose:

- `src/index.html`
- `src/styles.css`
- `src/app.js`

The backend preserves the existing browser contract:

- `GET /healthz`
- `GET /api/status`
- `POST /api/chat`
- static site served from `dist/`

### Backend

The old single-file server is gone. The new backend uses:

- **FastAPI** for routing and app lifecycle
- **Pydantic Settings** for env/config handling
- **httpx** for provider calls
- **rank-bm25** for corpus retrieval
- **uvicorn** for serving

Backend package layout:

- `fork_tales_api/app.py` — FastAPI app factory
- `fork_tales_api/settings.py` — env resolution and provider config
- `fork_tales_api/retrieval.py` — BM25 corpus index
- `fork_tales_api/service.py` — site loading, retrieval, live oracle calls, fallback behavior
- `fork_tales_api/schemas.py` — request/response models

## Local development

### 1. Install dependencies

```bash
cd /home/err/devel/projects/webring-sites/fork-tales-site
python3 -m venv .venv
source .venv/bin/activate
pip install -e '.[dev]'
```

### 2. Build the site snapshot

This still curates content from local source roots. By default it reads:

- `/home/err/devel/orgs/octave-commons/fork_tales`
- `/home/err/Music`

You can override those with:

- `FORK_TALES_SOURCE_ROOT`
- `FORK_TALES_MUSIC_ROOT`

Then build:

```bash
python build_site.py
```

### 3. Run the API

```bash
python server.py --root dist --host 127.0.0.1 --port 8794 --reload
```

Open:

- `http://127.0.0.1:8794`
- `http://127.0.0.1:8794/api/status`

## Provider configuration

For z.ai / GLM 5 Turbo:

```env
ZAI_BASE_URL=https://api.z.ai/api/paas/v4
ZAI_API_KEY=...
FORK_TALES_MODEL=glm-5-turbo
```

The backend also accepts OpenAI-compatible fallback names:

- `OPEN_HAX_OPENAI_PROXY_URL`
- `OPEN_HAX_OPENAI_PROXY_AUTH_TOKEN`
- `OPENAI_BASE_URL`
- `OPENAI_API_KEY`

## Container runtime

The runtime image serves the prebuilt `dist/` directory.

```bash
python build_site.py
docker compose up -d --build
```

Default bind:

- `127.0.0.1:8794 -> container :8080`

Useful runtime env:

- `FORK_TALES_PORT`
- `FORK_TALES_BIND_HOST`
- `FORK_TALES_CONTAINER_NAME`
- `FORK_TALES_MODEL`
- `ZAI_BASE_URL`
- `ZAI_API_KEY`

## Deploy flow

This project now follows the same familiar promotion pattern used elsewhere:

- feature branch -> **PR into `staging`**
- push to `staging` -> **deploy staging**
- `staging` -> **PR into `main` only**
- push to `main` -> **deploy production**

### Runtime conventions

- staging SSH host: `ussy3.promethean.rest`
- staging public host: `staging.fork.tales.promethean.rest`
- staging path: `~/devel/services/fork-tales-site-staging`
- staging compose project: `fork-tales-site-staging`

- production SSH host: `big.ussy.promethean.rest`
- production public host: `fork.tales.promethean.rest`
- production path: `~/devel/services/fork-tales-site`
- production compose project: `fork-tales-site`

### Deploy script

The remote deploy script does four things:

1. syncs this project to the remote runtime root
2. creates a build venv on the host
3. runs `python build_site.py` on the host so the live corpus is rebuilt from local content sources
4. runs `docker compose up -d --build`

Optional Caddy route re-apply now happens in deploy, not inside the app process.

## CI expectations

CI does **not** rebuild the real site corpus on GitHub runners, because the content sources are local workstation / host assets, not repo assets.

CI validates:

- backend imports
- API tests against fixture content
- shell script syntax
- promotion rules

The real site snapshot is rebuilt during remote deploy on the Promethean host.
