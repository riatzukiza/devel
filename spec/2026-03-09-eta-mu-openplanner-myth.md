# eta-mu x OpenPlanner: Memory, Myth, and Slop for the Ussyverse

## Signal

`part64` is not a pantheon simulator. It is a memory-bearing world.

OpenPlanner is the witness-lake.
eta-mu is the stage machinery that makes memory visible, navigable, and dramatic.
The inbox at `.ημ/` is the river mouth where raw offerings enter the world.
The shared Tailscale gateway is the public temple door.

## Universe

The useful cosmology is small.

- `Receipt River` remembers what entered the system.
- `Witness Thread` preserves chains of proof and consequence.
- `Anchor Registry` keeps drift from dissolving the work.
- `Gates of Truth` stop fantasy from rewriting evidence.
- `File Sentinel` watches the shape of change.
- `Change Fog` names ambiguity rather than pretending it is clarity.
- `Path Ward` keeps routes traversable.
- `Manifest Lith` binds artifact to declaration.
- `Core Pulse` is the felt center: the system is alive because it still answers.
- `Health Sentinels` keep compute pressure honest.

Everything else must justify itself by function, not vibe.

## Intention

OpenPlanner should hold the durable memory of:

- operator input
- muse events that materially change context
- presence utterances worth recalling
- security and crawler signals that deserve future retrieval

eta-mu should consume that memory back as:

- recall for presence replies
- searchable context for muses
- eventually, exported study packets and ring-ready narrative fragments

This makes the system one loop:

1. The workspace emits traces.
2. eta-mu shapes traces into world events.
3. OpenPlanner stores them as searchable witness.
4. eta-mu retrieves them as memory.
5. The ring receives slop that already knows what it came from.

## Presence Purge

Removed from the canonical field mythology because they did not carry clear operational purpose:

- `principle_good`
- `principle_evil`
- `principle_right`
- `principle_wrong`
- `state_dead`
- `state_living`
- bootstrap muses `stability` and `symmetry`

Not removed in this pass:

- `chaos_butterfly`

Reason: it is still structurally wired into the particle engine and render path. It is a runtime refactor target, not safe deadwood.

## OpenPlanner Role

OpenPlanner is the canonical long-memory layer.

- `eta-mu-user` events store operator gestures, queries, and direct inputs.
- `eta-mu-presence` events store purposeful utterances from operational presences.
- `eta-mu-muse` events store muse actions and significant emitted state transitions.

This means memory is no longer ornamental. It is queryable witness.
